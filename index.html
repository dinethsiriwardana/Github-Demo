GitHub 101

A complete beginner's guide to version control and collaboration with GitHub

Introduction to GitHub

GitHub is a web-based hosting service for version control using Git. It offers all of the distributed version control and source code management (SCM) functionality of Git as well as adding its own features.

GitHub provides a platform for millions of developers to collaborate on projects, track changes, manage code, and build software together. It's the world's largest host of source code and has become an essential tool for individual developers, open source projects, and businesses of all sizes.

Note: This guide assumes no prior knowledge of Git or version control systems. We'll start from the very basics and build up your understanding step by step.

Why use GitHub?

  • Version control: Keep track of changes to your code over time and revert to previous versions if needed
  • Collaboration: Work with other developers seamlessly on the same codebase
  • Code hosting: Store your code securely in the cloud
  • Project management: Track issues, plan work, and manage projects
  • Community: Engage with the largest developer community in the world

What You'll Learn

In this guide, you'll learn how to:

  • Create a GitHub account and set it up
  • Create your first repository
  • Understand basic Git commands and workflows
  • Collaborate with others using pull requests
  • Manage projects with issues and project boards
  • Use advanced GitHub features like Actions and Pages

Getting Started with GitHub

Creating a GitHub Account

Before you can use GitHub, you need to create an account:

  1. Visit github.com
  2. Click on "Sign up" in the top right corner
  3. Follow the prompts to create your account
  4. Verify your email address when prompted

Setting Up Git on Your Computer

To use Git locally, you need to install it on your computer:

Installation

For macOS:

brew install git

For Windows:

Download and install Git from git-scm.com

For Linux (Ubuntu/Debian):

sudo apt update && sudo apt install git

Initial Configuration

After installing Git, you should set up your identity:

git config --global user.name "Your Name" git config --global user.email "your.email@example.com"
Pro Tip: Use the same email address that you used to sign up for GitHub.

Understanding Repositories

A repository (or "repo") is like a project folder that contains all of your project's files and each file's revision history.

Creating Your First Repository

Step 1: Create a new repository on GitHub
  1. Click on the "+" icon in the top-right corner of GitHub
  2. Select "New repository"
  3. Enter a name for your repository
  4. Add an optional description
  5. Choose whether to make it public or private
  6. Select "Initialize this repository with a README"
  7. Click "Create repository"
Step 2: Clone the repository to your local machine

Cloning creates a local copy of the repository on your computer:

  1. On your GitHub repository page, click the green "Code" button
  2. Copy the URL shown (HTTPS or SSH)
  3. Open Terminal (Mac/Linux) or Git Bash (Windows)
  4. Run the following command:
git clone https://github.com/username/repository-name.git

Repository Structure

A typical GitHub repository includes:

  • README.md: A markdown file that explains the project
  • .gitignore: Specifies files Git should ignore
  • LICENSE: Defines how others can use your code
  • Source code files: The actual code for your project
  • Documentation: Additional information about your project

Git Basics

Git is a distributed version control system that forms the foundation of GitHub. Here are the essential Git commands you'll use regularly:

Command Description
git status Shows the status of files in your repository
git add [file] Stages a file for commit
git commit -m "message" Records changes with a message
git push Uploads local changes to GitHub
git pull Downloads changes from GitHub
git branch Lists branches in your repository
git checkout [branch] Switches to the specified branch

The Basic Git Workflow

  1. Pull the latest changes from the remote repository to ensure you're working with the most recent code:
    2. git pull origin main
  2. Create a new branch for your feature or fix:
    3. git checkout -b feature-name
  3. Make your changes to the files in your local repository
  4. Check the status to see what files have changed:
    5. git status
  5. Add your changes to the staging area:
    6. git add .
  6. Commit your changes with a descriptive message:
    7. git commit -m "Add new feature: feature name"
  7. Push your changes to the remote repository:
    8. git push origin feature-name

Understanding Commits

A commit is a snapshot of your repository at a specific point in time. Think of it as saving a version of your project that you can return to later.

Writing Good Commit Messages

A good commit message clearly communicates what changes were made and why. Follow these guidelines:

  • Be concise but descriptive (aim for under 50 characters)
  • Use the imperative mood ("Add feature" not "Added feature")
  • Include the area of code affected at the beginning
  • For complex changes, include a body after the summary line that explains why the change was made

Examples of good commit messages:

Fix navbar responsiveness on mobile devices
Add user authentication feature
Update README with installation instructions
Fix bug #123: Prevent form submission on empty fields
Important: Commit early and commit often! Small, focused commits are easier to understand, review, and revert if necessary.

Viewing Commit History

To view the commit history of your repository:

git log

For a more concise view:

git log --oneline

Working with Branches

Branches allow you to work on different features or fixes in isolation without affecting the main codebase.

What is a Branch?

A branch is a parallel version of your repository. By default, your repository has one branch called "main" (formerly "master"). When you create a new branch, you're creating a copy of the main branch as it was at that point in time.

Common Branch Operations

Creating a Branch

git branch branch-name

Creating and Switching to a New Branch

git checkout -b branch-name

Switching Between Branches

git checkout branch-name

Listing All Branches

git branch

Merging a Branch

git checkout main git merge branch-name

Deleting a Branch

git branch -d branch-name

Branching Strategies

A common branching strategy for teams is:

  • main/master: Contains stable, production-ready code
  • develop: Integration branch for features
  • feature/[name]: Individual features being developed
  • hotfix/[name]: Quick fixes for production issues
Note: The branch strategy can vary depending on team size and project needs. Find what works best for your situation.

Pull Requests

Pull requests (PRs) are a way to propose changes to a repository. They allow you to notify others about changes you've pushed to a branch, and request that someone review and merge your changes.

Creating a Pull Request

  1. Push your branch to GitHub:
    2. git push origin branch-name
  2. Go to your repository on GitHub
  3. Click on "Pull requests" tab
  4. Click the "New pull request" button
  5. Select your branch as the "compare" branch
  6. Review your changes and click "Create pull request"
  7. Add a title and description explaining your changes
  8. Click "Create pull request" to submit it

Pull Request Best Practices

  • Keep PRs small and focused on a single issue or feature
  • Write a clear description of what the PR does and why
  • Reference related issues with "#123" syntax
  • Add screenshots if there are visual changes
  • Request reviews from relevant team members
  • Respond promptly to feedback and make requested changes

Code Reviews

Code reviews are an essential practice where teammates examine each other's code changes to ensure quality, share knowledge, and catch potential issues.

Reviewing a Pull Request

  1. Go to the "Pull requests" tab in the repository
  2. Click on the PR you want to review
  3. Click on the "Files changed" tab to see the changes
  4. Add comments by clicking the "+" button that appears when hovering over a line
  5. Submit your review by clicking "Review changes" in the top right
  6. Choose to comment, approve, or request changes
  7. Add summary comments if needed and submit your review

Being a Good Reviewer

  • Be respectful and constructive in your feedback
  • Focus on the code, not the person
  • Ask questions instead of making demands
  • Praise good practices and clever solutions
  • Consider both technical correctness and readability
  • Be explicit about what needs to be changed versus what's just a suggestion
